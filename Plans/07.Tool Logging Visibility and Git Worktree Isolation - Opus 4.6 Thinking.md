# Tool Logging Visibility & Git Worktree Isolation

Fix two operational issues discovered during runs: (1) tool outputs (`[fs-tools]`, `[git]`, `[shell]`, `[GitHub]`, `[diagram]`, `[oauth-auth]`, `[workspace]`) are visible in the console but missing from `run.log`, and (2) parallel developer agents interfere with each other because they share a single git working directory — causing checkout races, "local changes would be overwritten" errors, and branches created from `master` instead of the project's system branch.

---

## Context & Current State

### Problem 1: Missing Tool Logs in `run.log`

The shared logger (`src/utils/logger.ts`) provides `getLogger()` which writes to both the console (with ANSI colors) and `run.log` (plain text via `appendToRunLog`). However, all tool files (`workspace-tools`, `git-tools`, `shell-tools`, `github-tools`, `diagram-tools`) and utility files (`oauth-auth.util`, `workspace`) use bare `console.log()` calls for their tagged output (e.g. `[fs-tools] write_file: src/index.ts`). These messages appear in the terminal but never reach the run log file, making post-run debugging incomplete.

### Problem 2: Git Checkout Races & Wrong Base Branch

The `executePRWorkflow()` function in `src/conductor/pr-workflow.ts` operates on the shared `workspacePath` directory for all branches. When the dispatcher runs multiple PR workflows in parallel (up to `MAX_CONCURRENT_DEVS` at a time), each workflow does:

```
git checkout <baseBranch>
git pull origin <baseBranch> --ff-only
git checkout -b <featureBranch>
```

Since all workflows share the same git working directory, this causes:

1. **Checkout races** — Workflow A checks out branch A, then Workflow B immediately checks out branch B, leaving Workflow A's agent on the wrong branch.
2. **"Local changes would be overwritten"** — One agent writes files but hasn't committed; another agent tries to `git checkout`, which git rejects because uncommitted changes would be lost.
3. **Branches from `master`** — The dev agent's prompt said "Switch to this branch using `git_checkout_branch`", which calls `git checkout -b <branch>` with `fromBranch` defaulting to `GIT_DEFAULT_BRANCH` (master). The branch was already created by the PR workflow from the system branch, but the agent re-created it from master.

---

## Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Tool logging mechanism | New `logToolAction()` in `logger.ts` | Single function that does `console.log` + `appendToRunLog(stripAnsi(...))`. Minimal change per tool file — just swap `console.log` → `logToolAction`. |
| Branch isolation | **`git worktree`** | Each branch gets its own working directory. Worktrees are a native git feature designed exactly for this: multiple branches checked out simultaneously without interference. No need for repo cloning or file copying. |
| Worktree location | `.worktrees/<branch-slug>/` under the git root | Keeps worktrees co-located with the repo. Added to `.gitignore`. Slug derived from branch name with non-alphanumeric chars replaced by `-`. |
| Worktree base | `origin/<baseBranch>` (fetched first) | Always uses the latest remote system branch, ensuring merged PRs from earlier batches are included in subsequent branches. |
| Agent prompt | "You are already on this branch. Do NOT create or switch branches." | Prevents agents from calling `git_checkout_branch` (which defaults to master). The worktree IS the branch — no switching needed. |
| Worktree cleanup | `finally` block in `executePRWorkflow` | Guarantees cleanup on both success and failure. Removes worktree directory and deletes the local branch ref. |
| Post-merge behavior | No local checkout/pull on main workspace | Old code did `git checkout baseBranch` + `git pull` after merge, which would disrupt other agents. With worktrees, the merge is via GitHub API only; the worktree is simply removed. |

**File:** `src/conductor/pr-workflow.ts`

Added imports and a helper function:

```ts
import * as path from 'path';
import * as fs from 'fs';

function findGitRoot(startPath: string): string {
    let dir = path.resolve(startPath);
    while (true) {
        if (fs.existsSync(path.join(dir, '.git'))) return dir;
        const parent = path.dirname(dir);
        if (parent === dir) throw new Error(`Not inside a git repository: ${startPath}`);
        dir = parent;
    }
}
```

### Step 10: Rewrite `executePRWorkflow` to Use Git Worktrees

**File:** `src/conductor/pr-workflow.ts`

Replaced the entire `executePRWorkflow` function body. Key changes:

**Worktree setup (replaces old branch creation):**
```ts
const gitRoot = findGitRoot(workspacePath);
const relativeWorkspace = path.relative(gitRoot, workspacePath);
const worktreeSlug = branchName.replace(/[^a-zA-Z0-9]+/g, '-');
const worktreeDir = path.join(gitRoot, '.worktrees', worktreeSlug);
const worktreeWorkspace = relativeWorkspace
    ? path.join(worktreeDir, relativeWorkspace)
    : worktreeDir;

// Clean up stale worktree from a previous failed run
if (fs.existsSync(worktreeDir)) {
    gitExec(gitRoot, `worktree remove "${worktreeDir}" --force`);
}
gitExec(gitRoot, `branch -D ${branchName}`);           // delete stale local branch
gitExec(gitRoot, `fetch origin ${baseBranch}`);         // get latest base
gitExec(gitRoot, `worktree add "${worktreeDir}" -b ${branchName} origin/${baseBranch}`);
fs.mkdirSync(worktreeWorkspace, { recursive: true });
```

**All internal operations use `worktreeWorkspace` instead of `workspacePath`:**
- `buildDevAgent(apiKey, entry, worktreeWorkspace)`
- `buildReviewerAgent(apiKey, reviewerEntry, worktreeWorkspace)`
- `gitExec(worktreeWorkspace, 'add .')`, `commit`, `push`, `diff`, etc.
- `writeArtifact({ workspacePath: worktreeWorkspace, ... })`

**Agent prompt changed:**
```
- Old: "Switch to this branch using git_checkout_branch or git_switch_branch before making changes."
+ New: "You are already on this branch. Do NOT create or switch branches — your workspace is isolated for this branch."
```

**Post-merge cleanup simplified:**
- Removed `git checkout baseBranch` + `git pull` on the main workspace (would disrupt other agents).
- Only deletes the remote branch via GitHub API after merge.

**`finally` block ensures cleanup:**
```ts
finally {
    if (fs.existsSync(worktreeDir)) {
        gitExec(gitRoot, `worktree remove "${worktreeDir}" --force`);
        log.info(`Cleaned up worktree: ${worktreeSlug}`);
    }
    gitExec(gitRoot, `branch -D ${branchName}`);
}
```

## How Worktree Isolation Works

```
Before (shared workspace — races):
┌─────────────────────────────────────────────┐
│  generated-projects/simplecalculator/       │  ← ALL agents share this
│    git checkout branchA  (agent 1)          │
│    git checkout branchB  (agent 2)  💥 RACE │
└─────────────────────────────────────────────┘

After (isolated worktrees — no races):
┌──────────────────────────────────────────┐
│  .worktrees/feature-us-001-.../          │  ← Agent 1 only
│    generated-projects/simplecalculator/  │
│    (on branch feature/us-001-...)        │
├──────────────────────────────────────────┤
│  .worktrees/feature-us-002-.../          │  ← Agent 2 only
│    generated-projects/simplecalculator/  │
│    (on branch feature/us-002-...)        │
└──────────────────────────────────────────┘
```

Each worktree is created from `origin/<systemBranch>` (e.g. `origin/project/simple-calculator`), never from `master`. After the PR is merged (or the workflow fails), the worktree is removed and the local branch is deleted.

---

## Risks & Mitigations

- **Disk space from worktrees** → Each worktree is a full checkout of the repo. Mitigated by `finally` block cleanup — worktrees are removed after each workflow completes, successful or not.
- **Stale worktrees from crashes** → If the process is killed mid-run, worktrees may be left on disk. The setup code checks for and removes stale worktrees before creating new ones. `git worktree prune` can also be run manually.
- **Agent still calls `git_checkout_branch`** → The updated prompt tells agents not to switch branches. If an agent ignores this and calls `git_checkout_branch`, it would try to create a new branch from `GIT_DEFAULT_BRANCH` inside the worktree — git would reject this since the worktree locks the current branch. The error is caught and logged.
- **`logToolAction` called before `setRunLogPath`** → If tools log before the run log path is set (during startup), `appendToRunLog` is a no-op (checks for null path). This matches existing logger behavior.
- **Worktree path collisions** → Branch names are slugified (non-alphanumeric → `-`). Two branches with names that slugify identically could collide. This is extremely unlikely given the naming convention (`feature/<story-id>-<description>`).
