# System Branches, Architect Simplicity & DBA Tag Fix

Enhance the git workflow so each system gets a dedicated `project/<system-name>` branch (never merging into main/master), simplify the architect's prompt to avoid over-engineering, unify the DBA log tag, and remove `generated-projects` from `.gitignore`.

---

## Context & Current State

Today, developer PRs target `main`/`master` directly, meaning all generated project output merges into the default branch. The architect agent tends to over-engineer simple systems (e.g. adding Kubernetes, Redis, Winston pipelines to a simple calculator). The DBA agent logs show inconsistent tags (`[DBA]` vs `[dba]`) due to the artifact writer using the lowercase agent ID.

**What changes:**
- `generated-projects/` is removed from `.gitignore` so output is tracked.
- Each system gets a long-lived **system branch** (`project/<system-name>`), created from `main`/`master`.
- Dev feature branches are created from the system branch; PRs target the system branch.
- After merge, dev feature branches are deleted (local + remote).
- Nothing auto-merges into `main`/`master`.
- The architect prompt gains proportionality rules to prevent over-engineering.
- The DBA log tag is unified to `[DBA]`.

---

## Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| System branch naming | `project/<system-name>` | Clear namespace, user-selected convention |
| Branch creation timing | During intake node | Earliest point where `systemName` and git root are known |
| State propagation | New `systemBranch` field in `ProjectState` | Flows through all downstream nodes without extra params |
| Architect fix | Prompt-level `<proportionality>` section | No structural change needed; prompt engineering is sufficient |
| DBA tag fix | Optional `tag` field in `ArtifactOptions` | Minimal change, doesn't affect other agents, extensible |

## Step-by-Step Implementation

### Step 1: Remove `generated-projects/` from `.gitignore`

**File:** `.gitignore`

Deleted the `generated-projects/` line.

### Step 2: Add `systemBranch` to Project State

**File:** `src/conductor/state.ts`

Added `systemBranch: Annotation<string>` with replace reducer and empty string default, between `outputPath` and `codebaseAnalysis`.

### Step 3: Update PR Workflow to Use `baseBranch`

**File:** `src/conductor/pr-workflow.ts`

- Added `baseBranch: string` to `PRWorkflowInput` interface.
- Removed `GIT_DEFAULT_BRANCH` import (no longer needed here).
- Step 1: Feature branches created from `baseBranch` instead of `GIT_DEFAULT_BRANCH`.
- Step 3: PRs target `baseBranch` instead of `GIT_DEFAULT_BRANCH`.
- Step 4: Diff computed against `baseBranch`.
- Step 5: After merge, switches back to `baseBranch`, then deletes the dev feature branch both locally (`git branch -D`) and remotely (`octokit.git.deleteRef`).

### Step 4: Update Dispatcher to Pass `baseBranch`

**File:** `src/agents/developers/dispatcher.ts`

- Added `baseBranch: string` parameter to `dispatchDevelopers()`.
- Passed `baseBranch` through to each `executePRWorkflow()` call.

### Step 5: Create/Checkout System Branch in Intake Node

**File:** `src/conductor/nodes.ts`

- Added `gitExec()` helper function (mirrors the one in `pr-workflow.ts`).
- In `intakeNode`, after git repo validation:
  - Derives system branch name: `project/${slugify(systemName)}`.
  - Checks if branch exists (remote/local).
  - **Greenfield:** Creates branch from default branch (or checks out if already exists).
  - **Maintain:** Checks out existing system branch, merges from default branch.
  - Pushes system branch to remote.
  - Stores `systemBranch` in returned state.
- In `developmentNode`: passes `state.systemBranch` as `baseBranch` to `dispatchDevelopers()`.

### Step 6: Update Team Leader Prompt

**File:** `src/agents/team-leader/team-leader.prompt.ts`

Updated `<branching_rules>` section 2 to clarify:
- All dev work targets the system branch (`project/<system-name>`), NOT main/master.
- Feature branches are created FROM the system branch.
- PRs are opened against the system branch.
- After merge, dev feature branches are deleted.

### Step 7: Add Proportionality Rules to Architect Prompt

**File:** `src/agents/architect/architect.prompt.ts`

Added `<proportionality>` section after `</critical_rules>`:
- Simple apps should not get Kubernetes, Redis, message queues, or complex observability.
- Docker is fine for any project; orchestrators only when spec demands scaling/HA.
- Caching layers only when spec mentions performance/traffic requirements.
- Prefer the simplest technology that satisfies requirements.

### Step 8: Fix DBA Log Tag

**Files:** `src/agents/_shared/artifact.ts`, `src/conductor/nodes.ts`

- Added optional `tag?: string` field to `ArtifactOptions` interface.
- `writeArtifact` now uses `opts.tag ?? \`[\${opts.agentId}]\`` for the logger tag.
- DBA's `writeArtifact` call in `nodes.ts` now passes `tag: '[DBA]'`.
