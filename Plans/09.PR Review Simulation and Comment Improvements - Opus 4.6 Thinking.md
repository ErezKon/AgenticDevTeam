# PR Review Simulation & Comment Improvements

Fix five issues observed in the basecalc run: enforce review hierarchy via prompt improvements, simulate PR review actions as structured GitHub comments (since all agents share one token), add "Opened By Agent X" to PR descriptions, deduplicate review comments across iterations, and add a PR-comment test to the git workflow integration test.

---

## Context & Current State

From the `basecalc-2026-07-23T15-23-38-429Z/run.log`:

1. **Wrong reviewer rank**: Principal FE's PR was reviewed by Senior FE — violates the rule "Principal → 2 other Principals" (`team-leader.prompt.ts` line 69).
2. **PR review comments fail to post**: GitHub returns `"Can not request changes on your own pull request"` and `"Path could not be resolved"` because all agents share one GitHub token/user.
3. **No author attribution**: PR descriptions don't mention which agent opened the PR.
4. **Review iterations are wasteful**: Reviewers re-raise the same comments every iteration (4-7 comments per reviewer × 5 iterations for the same PR).
5. **No comment-posting test**: `git-workflow.test.ts` tests branch/commit/push/PR-creation but not commenting.

---

## Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Review hierarchy enforcement | **Prompt-only** (improve TL prompt) | Per user preference — no hard-coded validation in dispatcher |
| Simulated PR actions | **Structured comment tags** via `octokit.issues.createComment()` | `createReview()` fails when author === reviewer. Issue comments work for any user. Tags like `[REVIEW: CHANGES_REQUESTED by Principal BE]` are machine-parseable. |
| Review deduplication | **Code-level dedup** in `pr-workflow.ts` review loop | Filter out comments whose `filePath + body-hash` already appeared in a prior iteration. Reduces noise and prevents infinite fix loops. |
| PR description author line | Prepend `**Opened by {Agent Name} ({agent-id})**` | First line of PR body, before Task Summary |
| Simulated reviewer request | **Issue comment** when PR is created | `"[REVIEW_REQUEST] {Author Agent Name} requested review from {Reviewer 1 Name} and {Reviewer 2 Name}"` |
| Comment format for review | **Issue comment** per review decision | `"[REVIEW: APPROVED by {Name}]"` or `"[REVIEW: CHANGES_REQUESTED by {Name}]\n\n{summary}\n\n### Comments\n- file:line — body"` |

### Step 1: Improve Team Leader Prompt — Reviewer Rank Rules

**File:** `src/agents/team-leader/team-leader.prompt.ts`

In the `<branching_rules>` section, strengthen the reviewer assignment rule (lines 66-71). Add explicit examples and a "NEVER" constraint:

```
    1. ASSIGN REVIEWERS based on developer rank:
       - Junior developer → assign 2 Senior developers as reviewers
       - Senior developer → assign 2 Principal developers as reviewers
       - Principal developer → assign 2 OTHER Principal developers as reviewers (never self-review)
       - NEVER assign a lower-rank developer to review a higher-rank developer's code.
         A Senior must NEVER review a Principal's PR. A Junior must NEVER review a Senior's or Principal's PR.
       - Reviewers must be from a RELEVANT domain (frontend reviewer for frontend code, etc.)
       - If only one reviewer of the right rank/domain exists, assign that one plus the closest match FROM THE SAME OR HIGHER RANK.
```

    `[REVIEW: ${statusTag} by ${reviewerEntry.name} (${reviewerEntry.id})] — iteration ${iteration}`,

### Step 4: Add Review Comment Deduplication

**File:** `src/conductor/pr-workflow.ts`

**4a.** Before the review loop (before line 368), create a `Set` to track seen comments:

```ts
const seenCommentKeys = new Set<string>();
```

**4b.** After a reviewer produces comments, filter out duplicates before recording:

```ts
function commentKey(c: { filePath: string; body: string }): string {
    // Normalize: lowercase file path + first 100 chars of body
    return `${c.filePath.toLowerCase()}::${c.body.slice(0, 100).toLowerCase()}`;
}
```

When collecting comments to pass to the dev agent for fixing (around line 494), filter:

```ts
const allComments = changesRequested.flatMap(r =>
    (r.output.comments ?? [])
        .map((c: any) => ({ ... }))
        .filter(c => {
            const key = commentKey({ filePath: c.file, body: c.comment });
            if (seenCommentKeys.has(key)) return false;
            seenCommentKeys.add(key);
            return true;
        })
);
```

**4c.** When there are zero new (non-duplicate) comments but reviewers still say `changes_requested`, auto-approve to break the loop:

```ts
if (allComments.length === 0) {
    log.info('All review comments are duplicates of prior iterations — treating as approved');
    prStatus = 'approved';
    break;
}
```

### Step 5: Add PR Comment Test to `tests/git-workflow.test.ts`

Add a new test case after the "should push the branch and create a PR" test:

```ts
it('should post a simulated review comment to the PR', async () => {
    // Post a simulated review comment
    const commentBody = '[REVIEW: APPROVED by Test Reviewer (test-reviewer)] — iteration 1\n\n**Summary:** Test review comment';
    const { data: comment } = await octokit.issues.createComment({
        owner: GITHUB_OWNER, repo: GITHUB_REPO,
        issue_number: createdPrNumber!, body: commentBody,
    });

    expect(comment.id).toBeGreaterThan(0);
    expect(comment.body).toContain('[REVIEW: APPROVED');

    // Verify the comment is readable
    const { data: comments } = await octokit.issues.listComments({
        owner: GITHUB_OWNER, repo: GITHUB_REPO,
        issue_number: createdPrNumber!,
    });
    const found = comments.find(c => c.body?.includes('[REVIEW: APPROVED by Test Reviewer'));
    expect(found).toBeDefined();
}, TIMEOUT);
```
